# Skill Reviewer

> 当前版本: v0.1.0（本 skill 以 git 管理版本，更新时同步修改此行并打 tag）

审查一个 skill 的质量，输出分级问题清单与修改建议。

**核心姿态：始终以「使用该 skill 的 Agent」的身份阅读被审 skill** —— 假装自己第一次拿到它、要完全依靠它完成任务。凡是这个视角下读不懂、走不通、会误导的，都是问题。

## 工作流程

### Step 1：确认审查目标

- 用户给出目录或 SKILL.md 路径 → 向上/向内定位到包含 SKILL.md 的 skill 根目录。
- 未给出目标 → 询问用户要审查哪个 skill。
- 同时确认一个事实供后续使用：**当前对话是否正在构建/迭代这个 skill**（是 → 检查项 2.12 启用；否 → 跳过 2.12）。这一点由自己根据对话历史判断，无需询问用户。

### Step 2：盘点文件与引用图

1. 递归列出 skill 根目录全部文件（排除 `.git/`、`__pycache__/`、`node_modules/`、虚拟环境目录）。
2. 从 SKILL.md 出发，沿 markdown 链接、路径引用、脚本内部的 import/路径，做**多跳可达性分析**，得到「可达文件集」与「孤立文件集」（供检查项 2.6 使用）。
3. **文件数 > 20**：用 AskUserQuestion 询问用户是否缩小审查范围。
   - 用户缩小 → 按用户指定范围继续。
   - 用户不缩小 → 按目录/主题拆分为若干分支，**并行派发子代理**，每个子代理负责一个分支；主会话自己保留需要全局视野的检查项（2.3 跨文件冲突、2.6 孤立文档、2.4 版本管理），最后汇总。

子代理 prompt 模板：

```
你是 skill 审查子代理。以「使用该 skill 的 Agent」视角审查以下文件：<分支文件列表>
被审 skill 根目录：<路径>
先完整阅读 <本 skill 根目录>/references/review-checklist.md，
然后对分支内每个文件执行清单中所有适用检查项。
只报告问题，不要修改任何文件。
返回问题列表，每条包含：检查项编号、优先级、位置(文件:行)、问题描述、建议修改。
涉及跨文件的疑点（表述冲突、引用缺失）单独标注「需主会话核对」并附上依据。
```

### Step 3：按清单审查

阅读 [references/review-checklist.md](references/review-checklist.md) 并执行其中全部检查项：

- **通用检查（2.1–2.12）**：对所有 skill 执行。
- **依赖与安装检查（3.1–3.4）**：仅当被审 skill 包含需要安装的工具/脚本、第三方依赖或登录流程时执行。

执行纪律：

- 每条问题必须落到**位置（文件:行）+ 问题描述 + 建议修改**，不接受"感觉不好"式的模糊反馈。
- 清单中标记「需用户判决」的检查项（2.6、2.7、2.10、2.11），发现问题时先**收集暂存**，全部审查完后在 Step 4 统一用 AskUserQuestion 询问，不要逐条打断。

### Step 4：输出报告并询问判决项

1. 按 [references/report-template.md](references/report-template.md) 的格式输出报告：P0（必须修复）→ P1（应当修复）→ P2（建议考虑），每级内按影响排序。
2. 将暂存的「需用户判决」问题集中列出，用 AskUserQuestion 逐类询问用户决定（孤立文档补引用还是删除、疑似违背机制处是否为环境特例、是否补充"绝对禁止"条目、反例范围是否收窄）。
3. 用户要求修改时才动手修改被审 skill；只要求 review 时，报告即为交付物。

### Step 5：（可选）增强 Review 模式

**默认不启用。** 仅当上述审查全部完成、且相应修改也已完成后，询问用户一次：是否需要进行增强 Review（联网对照领域现状，评估过时/不一致/遗漏/过度限制）。

用户同意 → 阅读并执行 [references/enhanced-review.md](references/enhanced-review.md)。

## 案例

**案例 1 — 小型 skill 直接审查**：用户说"帮我 review 一下 ~/projects/skills/foo"。确认根目录含 SKILL.md，共 6 个文件，无需拆分。通读后发现：SKILL.md 第 40 行引用 `scripts/convert.py` 但该文件不存在（2.2，P0）；`references/old-api.md` 无任何入链（2.6，P0，暂存待判决）。输出报告后用 AskUserQuestion 问用户 old-api.md 是补引用还是删除。

**案例 2 — 大型 skill 拆分审查**：被审 skill 有 35 个文件。询问用户是否缩小范围，用户选择全量。按 `references/`（18 个文件）、`scripts/`（12 个）、根目录+assets（5 个）拆成 3 个分支并行派发子代理，主会话自己核对跨文件冲突与孤立文档，合并三个分支的结果去重后输出统一报告。
