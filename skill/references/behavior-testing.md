# 实测验证模式

**前提**：默认不启用。常规审查完成后与增强 Review 一并询问用户，同意才执行。会派发多个子代理实跑任务，成本高于静态审查，先告知用户量级。

静态审查发现不了"description 写得没错但实际不触发"、"反例写了但压力下照样绕过"这类问题——本模式用实跑补上。三个子模式按成本从低到高排列，可单独或组合执行。

## 模式一：触发率抽测（成本低，优先做）

1. 基于被审 skill 的 description 与正文，构造 10–20 条 prompt 评测集，每条标注 `should_trigger: true/false`。**必须包含负样本**（skill 应保持沉默的请求：相邻领域、易混淆措辞），占比约 1/3；正样本要覆盖不同措辞（中英文、口语、文件类型/场景词）。
2. 每条 prompt 派子代理测试：仅提供被审 skill 的 name + description（模拟真实触发决策时可见的信息），询问"面对这个用户请求，你会加载这个 skill 吗？只答会/不会及理由"。每条重复 3–5 次看分布，不以单次为准。
3. 统计 under-trigger（正样本未触发）与 over-trigger（负样本被触发）比例，定位 description 措辞问题。最常见的高杠杆修复：把 API/技术术语改写为用户真实意图的措辞。

## 模式二：行为压力测试（RED / GREEN / REFACTOR）

适用于含约束、禁区、流程纪律的 skill——同时实测 2.10/2.11（反例是否挡住该挡的、是否误伤可行行为）。

1. **RED**：不提供被审 skill，给子代理一个真实任务，并**叠加多重压力**（时间紧迫 + 沉没成本 + 疲劳 + 后果严重），逐字记录它的自然选择与自我合理化说辞。单一压力压不出问题，必须组合。
2. **GREEN**：提供被审 skill 复跑同一任务，确认行为改为遵守，且子代理**能引用 skill 的具体章节**作为行为依据（引用不出来说明相关段落没起作用）。
3. **REFACTOR**：子代理会发明新的规避说辞。把每条说辞对应的显式反制建议补进报告（用户确认后写入 skill），复跑，循环到压不出新说辞为止。

## 模式三：Meta 反馈（成本极低，作为收尾必做）

让任何执行过被审 skill 的子代理（包括模式二的 GREEN 轮）追加回答："这个 skill 哪里可以写得更清楚？哪一步你犹豫或猜测了？"回答直接映射为文档缺口清单。

## 产出

实测结果并入 review 报告，单独成节：触发率数据（分正负样本）、RED/GREEN 行为对比、规避说辞与反制清单、meta 反馈缺口。修改建议仍走"用户确认后才改"的纪律。
