# Skill 审查清单

审查姿态：以「使用该 skill 的 Agent」身份通读全部文件。每条问题必须给出 **位置（文件:行）+ 问题描述 + 建议修改**。

优先级定义：

| 级别 | 含义 |
|------|------|
| P0 | 会导致使用失败或误导使用者，必须修复 |
| P1 | 明显降低质量或可维护性，应当修复 |
| P2 | 建议考虑，由用户裁量 |

标注「需用户判决」的项：发现问题先暂存，全部审查完后统一询问用户，不要逐条打断。

## 一、通用检查（所有 skill 均执行）

### 2.1 [P0] 工程信息泄露

检查是否存在只与"构建过程"相关、与"使用"无关的内容：

- 开发日志、设计讨论记录、TODO/FIXME 残留、作者自述
- 构建脚手架残留（init 模板的示例占位文件、示例目录）
- 面向人类维护者而非使用 Agent 的文档（README.md、CHANGELOG.md、INSTALLATION_GUIDE.md、QUICK_REFERENCE.md 之类）
- 测试过程说明、临时实验文件

动作：逐个列出，建议删除或移出 skill 目录。

### 2.2 [P0] 流程与工具完备性

以使用者身份把 SKILL.md 的主流程从头走一遍：

- 每一步引用的脚本、文件、命令是否真实存在于 skill 内
- 脚本的参数说明与脚本源码实际接受的参数是否一致
- 脚本内部 import / 读取的资源（模板、配置）是否在包内或有获取方式
- 步骤之间输入输出是否衔接（上一步产物是否是下一步声明的输入）
- 是否有断链："见 X 文档"但 X 不存在、"运行上面的命令"但上文没有命令

方法：对脚本至少做静态核对（读源码对照文档）；低成本可验证的做 dry-run（如 `--help`）。

动作：输出缺失/断链/不一致清单。

### 2.3 [P0] 前后文表述冲突

同一事项在不同位置的表述是否矛盾，**跨文件也要查**：

- 参数默认值、步骤顺序、文件路径、命名、版本号
- SKILL.md 与 references 之间、文档与脚本注释/`--help` 输出之间

动作：列出冲突对（两处位置+两种说法），并建议以哪处为准。

### 2.4 [P1] 版本管理与资产迁移

- skill 目录是否为 git 仓库（或位于 git 仓库内）？会迭代的 skill 需要版本管理。
- SKILL.md 是否标记了当前版本？
- 若 skill 的更新可能影响**用户资产**（用户按旧版 skill 在本地生成的配置、数据文件、模板、目录结构），是否有机制提示 Agent 检测版本差异并引导用户做资产迁移/更新，以对齐 skill 版本？

动作：无 git → 建议 `git init` 并打 tag；无版本标记 → 建议在 SKILL.md 顶部标注；有资产影响但无迁移机制 → 报 P1 并给出迁移提示的建议写法。

### 2.5 [P1] 表述冗余

- 同一信息是否在 SKILL.md 与 references 中重复维护
- 是否有大段解释模型本来就知道的常识
- 是否有可用 1 个简短例子替代的长段抽象描述
- SKILL.md 是否超过 500 行（应拆分到 references）

动作：指出可精简的段落，给出建议措辞或拆分方案。

### 2.6 [P0] 孤立文档/工具 —— 需用户判决

从 SKILL.md 出发做多跳可达性分析：沿 markdown 链接、正文中的路径引用、脚本内部 import/路径，标记所有**经多跳仍不可达**的文档、脚本、资产。

注意：一整组互相引用但整组都没有来自 SKILL.md 链路入口的文件（孤立文档集），同样算孤立。

动作：列出孤立文件清单 → 统一询问用户：补充引用，还是删除。

### 2.7 [P0] 违背常识/机制 —— 需用户判决

- 指令是否与平台机制冲突：要求 Agent 做其能力之外的事（常驻监听、跨会话记忆、读取不存在的 frontmatter 字段）、假设工具具有实际没有的能力
- 是否有事实性/技术性错误（API 用法、命令语法、格式规范）
- 流程逻辑是否自洽（先用后装、先读后生成之类的顺序颠倒）

动作：列出 → 询问用户判决（可能是用户环境特有的机制，不可武断判错）。

### 2.8 [P1] 案例缺失

对以下对象检查是否配有案例：

- 超过 3 个步骤的流程
- 复杂度较高或信息量较大的过程/方法

标准：每个场景至少 1–2 个案例，用最简洁明了的语言讲清**过程和结果**（输入 → 关键动作 → 产出）。案例是最优的说明方式，优于加长抽象描述。

动作：指出缺案例的流程，并给出建议的案例骨架。

### 2.9 [P1] 异常场景说明

枚举主流程可能失败的场景（网络不通、鉴权失效、文件缺失、格式不符、依赖版本漂移……），逐个判断：

- 该场景下环境/脚本反馈的报错信息，是否足以让使用 Agent 自行定位原因并解决？
- 不足以 → SKILL 中是否补充了「症状 → 原因 → 解法」的说明？

动作：列出报错语义不足且 SKILL 未覆盖的异常场景，建议补充说明或改进脚本报错。

### 2.10 [P2] 正例优先与关键禁区 —— 需用户判决

原则：**正例优先**。反例（"不要/禁止/绝不"）可能误伤——让模型回避反例相关但实际可行的行为。

检查：若整个 skill 中没有任何反例，对其中的**重要过程**收集候选风险点 → 询问用户：是否存在"绝对不可以"、"绝对不能"的操作需要写入 skill。

动作：询问用户，用户确认后才补充。

### 2.11 [P1] 反例严苛/污染检查 —— 需用户判决

对已有的每一条反例逐条评估：

- 表述范围是否过宽？（例：写"禁止使用网络"，实际只想禁"未经确认上传用户数据"）
- 是否会误伤正常、可行的行为？

动作：列出疑似过宽的反例及其可能误伤的场景 → 与用户确认收窄后的准确范围。

### 2.12 [P1] 隐式逻辑遗漏 —— 仅限构建对话

**适用条件**：仅当当前对话本身就在构建/迭代这个 skill 时启用；review 一个独立既有 skill 时跳过。（构建对话中 Agent 往往聚焦用户显式诉求，容易漏掉实体间的隐式逻辑。）

检查：枚举 SKILL 中说明的实体（工具、文件、状态、配置、角色），检查：

- 实体之间是否存在文中未写明、但使用时必须知道的隐式关联（例：脚本 A 的输出目录必须与配置 B 中的路径一致；状态文件 C 由步骤 2 创建、步骤 5 隐式依赖）
- 对话上下文中用户透露过、但没有落进 skill 的约束

动作：建议补充进 skill。

## 二、依赖与安装检查（仅当 skill 含需安装的工具/第三方依赖/登录时执行）

### 3.1 [P0] 独立安装指导

- 是否有独立的安装指导说明（单独的 reference 文件，而非散落在正文）？
- 工具集安装较复杂（多步骤、多依赖、需编译）时，是否附带安装脚本？

### 3.2 [P0] 环境隔离

安装应尽可能不影响系统环境：

- Python → venv / uv 等独立依赖环境，禁止直接 pip install 到系统环境
- Node.js → 项目局部 node_modules，避免 `npm install -g`
- 不修改系统级配置；确实无法隔离时，是否显式警告并说明影响

### 3.3 [P0] 状态/凭证 JSON

- skill 目录下是否约定了一个 **json 文件**，记录：依赖环境是否已装好、登录凭证
- CLI/脚本是否默认从该 json 读取凭证
- SKILL.md 是否声明默认前提：**每次使用 skill 时，默认所有工具已安装、用户已登录**，直接进入使用流程

### 3.4 [P1] 失败驱动安装

- 只有当工具调用报错、或返回鉴权失败时，才回头执行安装/登录流程
- SKILL.md 引用安装说明文档处，是否把这一点写成**读取条件**（如："仅当工具报错或鉴权失败时，才阅读 references/install.md"），而不是每次都读、或把安装放在流程第一步
