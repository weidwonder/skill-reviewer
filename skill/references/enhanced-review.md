# 增强 Review 模式

**前提**：常规审查已全部完成、相应修改也已完成，且用户在被询问后明确选择启用。未经用户同意不执行本模式。

目标：跳出 skill 文本本身，对照领域现状，发现常规审查发现不了的问题。

## Step 1：情报收集（可并行，建议派发子代理）

沿着被审 skill 的主题，从三个来源收集现状：

1. **GitHub 相似 skill**：搜索同领域的 skill / awesome 列表 / 同类 Agent 工具，重点看它们的流程设计、工具选型、参数约定与本 skill 的差异。
2. **互联网领域信息**：搜索该 skill 所依赖的工具、API、格式规范的最新版本与变化（弃用公告、breaking change、更优替代品）。
3. **用户主机上的相关项目**：仅当 skill 中存在对本机项目/路径的引用时，读取这些项目，理解 skill 在真实环境中的用法与假设。

收集以「充分理解领域现状」为度，不做无边界的发散搜索。派发子代理时，参照 SKILL.md Step 2 的子代理模板改写：把审查清单换成本文件，并要求每条结论附来源（URL 或本机路径）。

## Step 2：对照评估

基于收集到的现状，逐项判断被审 skill 是否存在以下四类问题，每条结论必须附来源依据（链接或本机路径）：

| 类别 | 判断标准 |
|------|----------|
| 过时 | 依赖的工具/API/格式已有新版本或已弃用，skill 仍按旧方式描述 |
| 不一致 | skill 的做法与领域主流实践明显相悖，且没有说明理由 |
| 条件分支或信息遗漏 | 领域中常见的场景/分支/参数，skill 未覆盖 |
| 过度限制 | skill 的约束在当前领域现状下已无必要，限制了能力 |

## Step 3：与用户讨论

将评估结果整理为更新意见清单（问题 → 依据 → 建议方案），与用户逐项讨论后再决定是否修改。**本模式只产出讨论方案，不直接修改 skill。**

## 案例

被审 skill 依赖某转换 CLI 并写死了 `--legacy-mode` 参数。Step 1 搜索发现该 CLI v2 官方公告已弃用此参数（附链接）；Step 2 将其归类「过时」；Step 3 产出建议：升级安装说明至 v2 并替换参数写法，交用户确认后再改。
